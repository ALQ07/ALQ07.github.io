## 前言

最近一直在做关于帧同步的一些研究，所以就想写一些我对帧同步的理解吧。

## 正文

## 房间系统

一般来说，在两个人或者多人联机游戏中，都会有一个房间系统来管理游戏中玩家的加入、退出、匹配等操作。

首先，当玩家加入房间之后，服务器会为该玩家分配一个唯一的玩家ID（PlayerID），并将该玩家加入到房间中。同时，服务器会取得玩家的信息，当玩家点击按钮开始游戏之后，玩家会向服务器请求开始游戏，然后服务器会将房间内玩家的初始信息整合然后转发给所有客户端，客户端会根据服务器转发的信息来初始化游戏环境。

## 服务器驱动

开始游戏后，服务器会每隔一段时间向客户端推送消息，如果说有玩家做了同步操作，那么服务器就会推送同步操作给所有的玩家，如果没有同步操作，那么服务器就推送一条空的数据给客户端，客户端根据有无同步操作执行相应的游戏逻辑。

### 服务器推送的同步数据有什么？

一般来说，服务器推送的同步数据会是一个数组，结构如下：

```TypeScript
    ['f', frameIndex, actions]
```

1. **消息类型标识 (`'f'`)**：
   - 类型：字符串 `'f'`。
   - 作用：告诉客户端这是一个正常的“帧数据包”（区别于 `'r'` 追帧包或 `'p'` P2P信令）。

2. **帧序号 (`frameIndex`)**：
   - 类型：数字类型（Integer）。
   - 作用：表示这是第几帧的数据（例如第 100 帧）。客户端用它来检查是否漏帧（如果收到 102 但当前是 100，说明漏了 101，会触发 `'r'` 请求重发）。

3. **操作指令列表 (`actions`)**：
   - 这是一个包含该帧内所有玩家操作的**数组**。
   - 数组中的每个元素是一个对象，包含该帧发生的具体事件，例如：
     - `initPlayer`：初始化玩家信息。
     - `add`：添加单位（如召唤物）。
     - `focus`：焦点目标改变。
     - `player_move`：玩家移动指令（包含方向、摇杆力度等）。
     - `cmd`：其它通用指令（如释放技能）。
   - **关键点**：如果这一帧没有任何人操作，这个列表可能是空的，或者只包含空指令，但帧数据包依然会发送，以驱动游戏时钟向前走（Heartbeat Frame）。

## 心跳检测

在游戏中，客户端会启动一个定时器，每隔一段时间向服务器发送一次ping消息，服务器收到ping消息后会返回一个pong消息。客户端根据服务器返回的pong消息来判断服务器是否在线，服务器根据客户端发送的ping消息来判断客户端是否在线。如果客户端断网断线，那么服务器会在一段时间内没有收到客户端的消息，就会判断客户端掉线。

## 断线重连的追帧逻辑

当客户端重连成功之后，客户端当前本地是第100帧，而服务器正常广播了第200帧的数据，那么客户端就会判定帧丢失，于是客户端会向服务器发送一个请求，告诉服务器“我卡在第 100 帧了，请把后面的发给我”。服务器收到请求后，将 101~200 帧的数据打包发回。

## 如何处理服务器返回的一大波追帧数据

当收到数据时，它不会立即执行逻辑，而是将这一大波帧数据 按顺序 塞入 frameList 队列中。设置 _isRestore = true ，告诉主循环“现在处于追帧模式”。

```TypeScript
    this._isRestore = this.frameList.length > 0; // 标记开始追帧
```

执行策略为：它不会一帧一帧地慢慢跑，而是在一个渲染帧内执行多次逻辑帧。

```TypeScript
// 计算本次要处理多少帧：取积压的一半，但受 maxRestoreFrames 限制
let max = Math.ceil(this.frameList.length / 2);
if (this._isRestore) {
    // FightConfig.maxRestoreFrames 默认为 600
    max = Math.min(max, this._context.FightConfig.maxRestoreFrames);
}

// 循环执行 max 次逻辑帧
for (let i = 0; i < max; i++) {
    let data = this.frameList.shift(); // 取出一帧数据
    this.onUpdate(data[0], data[1]);   // 执行这一帧的逻辑
}
```

上面这段代码中使用了一个动态策略： Math.ceil(this.frameList.length / 2) 。
- 这意味着如果你落后了 1000 帧，第一帧会追 500 帧，第二帧追 250 帧……
- 效果 ： 起步极快，越接近目标越平滑 。这能让玩家迅速看到最新画面，同时在快追上时减少计算压力，避免卡顿。
- 上限保护 ： maxRestoreFrames = 600 防止了一次性计算太多导致浏览器卡死（UI失去响应）。
